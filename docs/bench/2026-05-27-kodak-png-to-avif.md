## Інтерпретація

- **Розмір — практично паритет.** Різниця −0.3% по сумі — у межах шуму AVIF-кодера. Per-file: 12 файлів у Bun.Image менші (до −6.0%), 12 — більші (до +11.3%). На kodim22/14/05/24 (фото з низьким частотним вмістом) Bun.Image видає менші файли; на kodim20/19/09/16 (з більшою візуальною складністю) — більші. Тобто rate-distortion-курва двох кодерів перетинається — жоден не «кращий за байтами», обидва дають середній результат у однаковій корзині.

- **Швидкість — однозначний win Bun.Image.** Медіана 55 ms vs 505 ms — **9.23× швидше**. На macOS arm64 це Apple ImageIO з hardware acceleration через ANE/Media Engine; sharp/libavif йде через CPU SIMD. Різниця стабільна по всіх 24 файлах (8.06–10.31×).

- **Якість — sharp злегка кращий.** ΔSSIM −0.0038 (≈0.4% від абсолютного значення), ΔDSSIM +0.0017. Це чутно тільки порівнянням side-by-side; для UI-AVIF / web-thumbnails — undetectable.

## Кому що рекомендується

| Use case                                | Рекомендація                                                              |
| --------------------------------------- | ------------------------------------------------------------------------- |
| Photo AVIF для web (thumbnails, gallery) | **Bun.Image** — паритет розміру, 9× швидше, втрата якості невідчутна       |
| Archive / max-quality AVIF              | sharp — стабільно трохи краща метрика SSIM/DSSIM при тому ж розмірі       |
| Batch processing (CI/CD, build-pipeline) | **Bun.Image** — економія wall-clock 9× для AVIF домінує                   |

На фотокорпусі (Kodak) Bun.Image AVIF — практичний default. Поведінка на UI-/flat-graphics корпусах інша (див. `2026-05-26-bun-image-vs-sharp.md`, секція «Другий корпус: реальні UI-PNG»), там Bun.Image дає +38% розміру — фотокорпус і UI-корпус ведуть себе різно через відмінності rate-distortion ImageIO/libavif на різних типажах контенту.
