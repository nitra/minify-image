## ADR Методологічно коректний бенчмарк Bun.Image vs sharp — Kodak, SSIM/DSSIM, parity-параметри
**Контекст:** Попередній ADR `20260526-054228-bun-image-benchmark.md` дійшов висновку «sharp залишається», але мав три методологічні помилки: PNG тестувався без `palette: true` (що дає основну економію), JPEG — без `mozjpeg`/`progressive`, якість порівнювалась лише розміром без SSIM/DSSIM.
**Рішення/Процедура/Факт:** Побудовано незалежний harness у `bench/` (не workspace): `download-corpus.mjs` завантажує 24 Kodak PNG (768×512); `codecs/sharp.mjs` і `codecs/bun-image.mjs` — adapter-и з однаковою сигнатурою `encode(buf, fmt) → Promise<Uint8Array>` і параметрами-дзеркалами (`palette: true`, `progressive: true`, `quality: 40`); `quality.mjs` — SSIM через `ssim.js` + DSSIM через `dssim` CLI; `micro.mjs` — N=10, відкидаємо перший прогін, median+p95; `e2e.mjs` — 3 end-to-end CLI прогони з очищенням кешу; `report.mjs` → `docs/bench/`. Результати: Bun.Image на Kodak 9.23× швидший для AVIF, 1.95× для PNG, паритет розміру; на реальному UI-корпусі (lumik-ru/bono) PNG −38%, але AVIF +38% розміру.
**Обґрунтування:** Apples-to-apples parity параметрів є необхідною умовою чесного codec-порівняння; SSIM/DSSIM потрібні бо менший файл при гіршій якості не є виграшем; Kodak — академічний стандарт для image-codec бенчмарків. Попереднє рішення «sharp залишається» ґрунтувалось на некоректних даних.
**Розглянуті альтернативи:** тестовий корпус з `~/www/nitra/` (відхилено — lint-image вже оптимізував усі файли, нема необроблених великих PNG); butteraugli замість DSSIM (відхилено — зайва складність без додаткової цінності над SSIM+DSSIM).
**Зачіпає:** `bench/` (весь каталог), `docs/bench/2026-05-26-bun-image-vs-sharp.md`, `docs/bench/2026-05-27-kodak-png-to-avif.md`

## ADR Міграція @nitra/minify-image з sharp на Bun.Image, видалення GIF — v4.0.0
**Контекст:** Бенчмарк показав що на macOS arm64 з Bun 1.3.14 (`Bun.Image.backend = system`, Apple ImageIO) `Bun.Image` стискає PNG/JPEG/AVIF у 2–9× швидше за sharp при паритеті або кращому розмірі (крім UI AVIF де +38%). `sharp` — нативна бінарка ~150 MB в deps, вимагає компіляції при `npm install`. `Bun.Image` — вбудований у runtime, нульові додаткові залежності.
**Рішення/Процедура/Факт:** У `npm/src/index.js` блок `compressors` замінено: `sharp(buf).png({compressionLevel:9, effort:10, palette:true})` → `new Bun.Image(buf).png({compressionLevel:9, palette:true})`; аналогічно для `jpeg`, `avif`, `webp`. GIF-гілка, `compressors.gif`, `.gif` у `EXTENSIONS_ACCEPTED`/`EXTENSIONS_SKIP`/glob-patterns — видалені. Імпорт `import sharp from 'sharp'` видалено. `npm/package.json`: `"version": "3.6.0"` → `"4.0.0"`, `"sharp"` прибрано з `dependencies`, `"engines"` змінено з `"node": ">=22"` на `"bun": ">=1.3.14"`, `"description"` оновлено. `demo/test/files/minified.gif` видалено з fixtures. CHANGELOG — нова секція `4.0.0` з Breaking/Changed/Removed.
**Обґрунтування:** Повне видалення sharp = нульова native-binary deps-tree; виграш 9× на AVIF і 2–4× на PNG/JPEG підтверджений методологічно коректним бенчмарком. UI AVIF +38% прийнятний — обсяг AVIF у target-проектах 100–500 KB сумарно, не критично. GIF: 87% файлів < 10 KB (loaders — стискання беззмістовне), унікальних великих GIF ~7 штук (Bono-анімації) — недостатньо щоб тримати 150 MB native binary. Major bump семантично правильний: breaking change (GIF dropped, вимога bun замість node).
**Розглянуті альтернативи:** гібрид (Bun.Image для PNG/JPEG, sharp для AVIF+GIF) — відхилено, бо UI AVIF різниця прийнятна а dep-tree-win важливіший; GIF через `gifsicle-wasm` — відхилено (зайва dep, Bono-анімації міняються рідко).
**Зачіпає:** `npm/src/index.js`, `npm/package.json`, `npm/CHANGELOG.md`, `demo/test/files/` (видалено `minified.gif`), `demo/test/run.test.js`

## Knowledge Bun.Image API — обмеження та поведінка на Bun 1.3.14 (macOS arm64)
**Контекст:** При написанні codec-адаптера для бенчмарку та production-коду виявлено неочевидні властивості `Bun.Image` API, що впливають на fairness порівнянь і вибір параметрів.
**Рішення/Процедура/Факт:** (1) `Bun.Image.backend === 'system'` на macOS — це Apple ImageIO + ANE/Media Engine; на Linux backend інший, результати несумісні. (2) Опції `mozjpeg: true` і `effort: 10` silently ігноруються — API приймає без помилки, але не застосовує. (3) GIF encoder відсутній: `Bun.Image` не має методу `.gif()`, декодування GIF можливе, кодування — ні. (4) AVIF поведінка залежить від типу контенту: на фото (Kodak 768×512) — паритет розміру зі sharp `quality:40`; на UI flat-контенті (lumik-ru bg PNG) — +38% більший файл. (5) `Bun.Image` instance не реюзабельний між кадрами — кожен encode вимагає `new Bun.Image(buf)`.
**Обґрунтування:** Виявлено empirically під час probe-скриптів та micro-bench прогонів.
**Розглянуті альтернативи:** не обговорювались.
**Зачіпає:** `bench/codecs/bun-image.mjs`, `npm/src/index.js`, будь-який код що використовує `Bun.Image` для production image processing
