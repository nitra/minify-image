## ADR Bun.Image vs sharp — методологічно коректна переоцінка (Kodak + реальний UI-корпус, SSIM/DSSIM, palette-паритет)

**Контекст:** Попередній ADR (`20260526-054228`) відхилив Bun.Image як заміну sharp, але мав методологічні вади: PNG тестувався без `palette:true` (яка дає основну економію), JPEG — без `mozjpeg`/`progressive`, корпус — лише 23 власних активи без стандартизованого test suite. Потрібна була повторна оцінка з виправленими умовами.

**Рішення/Процедура/Факт:** Створено benchmark harness у `bench/`: завантажувач Kodak-24 corpus (`download-corpus.mjs`), adapters `codecs/sharp.mjs` та `codecs/bun-image.mjs` з paritied параметрами (однакові `quality`, `compressionLevel`, `palette`, `progressive` на обох — silently ignored на Bun.Image зафіксовано явно), модуль якості `quality.mjs` (SSIM через `ssim.js` + DSSIM через `dssim` CLI), micro-bench (`micro.mjs`, N=10 median+p95), e2e-CLI fork `e2e-cli-bun-image.mjs`, e2e orchestrator `e2e.mjs` (N=3 run wall-clock), report generator `report.mjs` → `docs/bench/2026-05-26-bun-image-vs-sharp.md`. Проведено два корпуси: Kodak 24 photo-PNG + реальний UI-корпус (lumik-ru backgrounds + bono logos, 25 PNG, 9.8 MB).

Результати:
- **Kodak (photo):** Bun.Image PNG −20% розміру 2×, JPEG ≈ паритет, AVIF ≈ паритет, WebP ≈ паритет; E2E: Bun.Image 8.94× швидший, output −19%
- **UI-корпус (flat graphics):** PNG −38% 4.33×, JPEG +12% але SSIM +0.133 (вища якість), **AVIF +38% (програш)**, WebP паритет
- **GIF:** 196 файлів у `~/www`, але 87% < 10 KB (лоадери); 14 файлів > 1 MB — фактично 5–7 унікальних Bono-анімацій; Bun.Image не має GIF encoder взагалі
- `Bun.Image.backend === 'system'` на macOS = Apple ImageIO + ANE hardware acceleration; Linux-результати невідомі

**Обґрунтування:** Гібридна стратегія рекомендована: PNG/JPEG/WebP → Bun.Image (швидше, якісніше або не гірше); AVIF → sharp (на UI-контенті Bun.Image дає більший файл); GIF → sharp або окремий `gifsicle` CLI (sharp як dep прийнятний лише якщо потрібен для AVIF; якщо AVIF на target — photo-only, sharp взагалі можна прибрати). Повне видалення sharp блокується: (1) GIF encoder відсутній у Bun.Image; (2) AVIF якість залежить від типу контенту; (3) Linux backend не перевірено.

**Розглянуті альтернативи:** (1) Повна заміна sharp → Bun.Image — відхилено через GIF + AVIF на UI + Linux невизначеність. (2) Залишити sharp без змін — відхилено як неоптимально (8–9× уповільнення без потреби). (3) Гібрид (рекомендовано) — Bun.Image для PNG/JPEG/WebP, sharp для AVIF/GIF. (4) Виключити GIF-підтримку з CLI, GIF стискати одноразово через `gifsicle` бінарку — обговорено як довгостроковий варіант для повного видалення sharp.
