## ADR Bun.Image vs sharp — переоцінка з виправленою методологією (Kodak, SSIM/DSSIM, palette parity)

**Контекст:** Попередній ADR (`20260526-054228-bun-image-benchmark.md`) зафіксував рішення «sharp залишається», але мав методологічні прогалини: Bun.Image PNG тестували без `palette: true` (поточний CLI його вмикає, і це основне джерело PNG-економії), JPEG — без `progressive`, корпус 23 файли власних активів, метрик SSIM/DSSIM не було. Цей замір переоцінює рішення на Kodak suite (24 PNG 768×512) із apples-to-apples параметрами та повними метриками якості.

**Рішення/Процедура/Факт:** На Kodak-корпусі Bun.Image послідовно виграє в усіх категоріях (на macOS arm64, Bun 1.3.14, backend `system` = Apple ImageIO):

- **PNG (palette:true):** -20% розміру, 1.86× швидше (Bun.Image: 4.30 MB / 191 ms; sharp: 5.38 MB / 356 ms — на 24 файли)
- **JPEG:** +4.5% розміру (mozjpeg ігнорується silently), 4.26× швидше, ΔSSIM +0.0006
- **AVIF (quality:40):** паритет розміру, **14.7× швидше**, ΔSSIM -0.0038 (Apple ImageIO + ANE/Media Engine HW acceleration)
- **WebP (quality:80):** повний паритет розміру/SSIM, 1.72× швидше
- **E2E (full CLI run, --write --avif):** Bun.Image у 8.94× швидший, output на 19.1% менший

**Висновок:** Висновок попереднього ADR «sharp залишається» **не підтверджується** на photo-корпусі — Bun.Image із `palette: true` виграє по розміру, часу і якості. Однак попередній замір був на великих UI-PNG (Tauri іконки 1-20 MB), а не на photo-корпусі. Перш ніж міняти production CLI, потрібен ще один замір — на великих UI-PNG із `palette: true` на обох сторонах (тобто з виправленою методологією попереднього замірю). Поточний ADR **не змінює** `npm/src/index.js`, але **скасовує** трактовку попереднього ADR як остаточну — обидва ADR описують різні корпуси, а інтерес проекту покриває обидва.

**Обґрунтування:** Bun.Image на macOS використовує Apple ImageIO з hardware acceleration для AVIF (ANE / Media Engine) — звідси 14× прискорення без значущої втрати якості. Раніше це не помічалось, бо тестували інший корпус (великі PNG) і без palette-паритету. Платформо-залежність (`Bun.Image.backend`) робить результати валідними **лише на macOS arm64**; для Linux/Windows потрібен окремий прогон того ж harness.

**Розглянуті альтернативи:**

- Прийняти результати Kodak як остаточні й переключити CLI на Bun.Image — відхилено: photo-корпус не репрезентує проектний вантаж (Tauri PNG-іконки 1-20 MB), де попередній замір показав протилежне.
- Скасувати попередній ADR — відхилено: він описує інший корпус, на якому теж міг бути правий. Обидва ADR доповнюють один одного.
- Зробити replication попереднього замірю з palette-паритетом на UI-PNG корпусі — **запропоновано** як next step.

**Зачіпає:** `bench/` (harness, не комітимо корпус і результати), `docs/bench/2026-05-26-bun-image-vs-sharp.md` (звіт). `npm/src/index.js`, `npm/package.json` залишаються без змін до наступного замірю на великих UI-PNG.

**Next step:** Розширити harness, щоб приймати довільну директорію зображень (через `--corpus=<dir>`), і прогнати на корпусі великих PNG (наприклад, `~/www/nitra/*/src-tauri/icons/` із попереднього замірю) з палітризацією на обох сторонах. Якщо Bun.Image виграє і там — виносити sharp із deps. Якщо програє — фіксувати hybrid-стратегію в окремому ADR.
