## ADR Benchmark workspace Bun.Image vs sharp

**Контекст:** Пакет `@nitra/minify-image` використовує `sharp` для стискання JPEG/PNG/GIF/AVIF. Bun 1.3.14 ввів нативний `Bun.Image` API, що може бути швидшим і не потребує нативного модуля. Потрібні об'єктивні дані перед ухваленням рішення про міграцію.

**Рішення/Процедура/Факт:**

- Створено окремий workspace `bench/` (з власним `package.json`, доданий до `workspaces` у кореневому `package.json`)
- Корпус — Kodak 24-image suite (PNG lossless) + 3 анімовані GIF + 3 SVG; lazy-download в `bench/corpus/` при першому запуску
- Метрики: `encode_ms` (wall-clock, без I/O), `orig_bytes`/`out_bytes`/`ratio`, `ssim` (через `ssim.js`, лише для lossy форматів)
- Обидва backend-и (`drivers/sharp.js`, `drivers/bun-image.js`) викликаються послідовно (не паралельно) з однаковими параметрами якості (JPEG=80, AVIF=50)
- `Bun.Image` не має `gif()` encoder у 1.3.14 → драйвер повертає `null`, метрики записуються як `N/A`
- Результати: `bench/results/YYYY-MM-DD.json` + `.csv` + ASCII-таблиця медіан у stdout

**Обґрунтування:** Ізольований workspace не торкається production-коду (`npm/src/index.js`). Послідовний прогін виключає contention між libuv thread pool (sharp) і нативними Bun threads. SSIM дає чесне порівняння lossy-форматів — менший файл при гіршій якості не є перемогою.

**Розглянуті альтернативи:**

- Один скрипт з двома driver-функціями — відкинуто: помилка в одному driver зупиняє все
- `bun bench()` API — відкинуто: призначений для мікробенчмарків CPU, не підходить для file-write + SSIM

**Зачіпає:** `bench/package.json`, `bench/corpus.js`, `bench/drivers/sharp.js`, `bench/drivers/bun-image.js`, `bench/metrics.js`, `bench/run.js`, кореневий `package.json` (workspaces)

## Knowledge Bun.Image 1.3.14 — обмеження форматів

**Контекст:** При реалізації `bench/drivers/bun-image.js` перевірено доступні методи `Bun.Image.prototype` у Bun 1.3.14.

**Рішення/Процедура/Факт:** `Bun.Image` підтримує кодери: `png()`, `jpeg()`, `webp()`, `avif()`, `heic()`. GIF encoder (`gif()`) **відсутній**. SVG не підтримується взагалі. Для цих форматів driver повинен повертати `null`, а не кидати помилку.

**Обґрунтування:** Протоколювати `N/A` замість винятку — єдиний спосіб зберегти однорідну структуру результатів і зробити GIF-рядки частиною фінальної таблиці (щоб явно показати, де Bun.Image відпадає).

**Розглянуті альтернативи:** не обговорювалися

**Зачіпає:** `bench/drivers/bun-image.js`, будь-яка майбутня міграція `npm/src/index.js` на `Bun.Image`

## Knowledge ssim.js — типи вхідних даних несумісні з sharp raw()

**Контекст:** При написанні `bench/metrics.js` виявлено, що `ssim.js@0.6.2` (встановлений у проєкті) оголошує тип `ImageData.data` як `Uint8ClampedArray | Float64Array`, але `sharp().raw().toBuffer()` повертає звичайний `Buffer`/`Uint8Array`.

**Рішення/Процедура/Факт:** Для передачі буфера в `ssim()` потрібно явно конвертувати: `new Uint8ClampedArray(data.buffer)` замість `new Uint8Array(data.buffer)`. Без цього TypeScript/ESLint з `@antfu/eslint-config` (тип `lib`) генерує помилку `@typescript-eslint/prefer-readonly-parameter-types` з хибним номером рядка, що ускладнює діагностику.

**Обґрунтування:** ESLint звітує `22:25` (рядок `return null`), але справжня причина — невідповідність типу аргументу `ssim()`. Це відомий pattern помилки TypeScript-aware ESLint: неправильне місце помилки при type inference failure.

**Розглянуті альтернативи:** не обговорювалися

**Зачіпає:** `bench/metrics.js` — функція `toRawRGB()`
