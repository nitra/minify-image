## ADR Bun.Image vs sharp — замір на реальних корпусах з коректною методологією (Kodak + UI-PNG)

**Контекст:** Попередній ADR (`20260526-054228-bun-image-benchmark.md`) зробив висновок «Bun.Image не рівноцінна заміна», але мав методологічні помилки: PNG тестувався без `palette: true`, JPEG без `mozjpeg`/`progressive`, без SSIM/DSSIM, лише на 23 власних активах. Ця сесія переробила замір apples-to-apples.

**Рішення/Процедура/Факт:** Побудовано harness `bench/` з адаптерами `bench/codecs/sharp.mjs` і `bench/codecs/bun-image.mjs`, downloader Kodak suite (24 PNG), модуль якості `bench/quality.mjs` (SSIM via `ssim.js` + DSSIM via `dssim` CLI), micro-bench `bench/micro.mjs` (N=10, median+p95), e2e-форк `bench/e2e-cli-bun-image.mjs`, генератор звіту `bench/report.mjs`. Паралельно проведено UI-замір на 25 реальних PNG з `lumik-ru/static/bg` + `bono/loyalty/assets`.

**Обґрунтування:** Bun 1.3.14 на macOS використовує `Bun.Image.backend === 'system'` (Apple ImageIO + ANE/Media Engine), що дає апаратне прискорення. Результати розходяться між типажами:

| Формат | Kodak (photo) | UI-PNG (flat) |
|---|---|---|
| PNG | −20%, 1.95× faster | −38%, 4.33× faster |
| JPEG | −10%, 2.7× faster | +12%, 2.87× faster (але SSIM +0.133 — якісніший) |
| AVIF | −10%, 7.6× faster | **+38%, 7.13× faster** — програш у розмірі |
| WebP | паритет | паритет |
| E2E wall-clock | — | 8.94× faster, −19% total size |

**Розглянуті альтернативи:** Не обговорювались — замір-дослідження, а не вибір між підходами.

**Зачіпає:** `bench/` (новий каталог), `docs/bench/2026-05-26-bun-image-vs-sharp.md`, `docs/adr/_inbox/20260526-224541-bun-image-revisited.md`; рішення про зміну `npm/src/index.js` відкладено до Linux-replication

## ADR Гібридна стратегія кодеків: Bun.Image для PNG/JPEG/WebP, sharp для AVIF і GIF

**Контекст:** За результатами замірів Bun.Image на UI-контенті дає AVIF-файли на 38% більші за sharp (libavif vs Apple ImageIO). GIF encoder у Bun.Image відсутній. Водночас PNG/JPEG/WebP у Bun.Image або кращі або паритет за розміром, і в 2–4× швидші.

**Рішення/Процедура/Факт:** Рекомендована архітектура для `npm/src/index.js`: PNG → `Bun.Image` (palette, compressionLevel:9), JPEG → `Bun.Image` (quality:75, progressive:true), WebP → `Bun.Image` (quality:80), AVIF → `sharp` (quality:40, libavif), GIF → `sharp` (unchanged). `sharp` залишається dep, але активно лише для AVIF і GIF. Зміни в `npm/src/index.js` відкладено до Linux Docker replication (на macOS `Bun.Image.backend === 'system'`; на Linux backend інший, результати невідомі).

**Обґрунтування:** AVIF через Apple ImageIO слабко стискає flat UI-контент — очевидно алгоритм оптимізований під photo. GIF: 87% GIF-файлів у `~/www` < 10 KB (loaders), лише ~7 унікальних > 1 MB (Bono-анімації). Тримати весь sharp лише для GIF економічно сумнівно, але поки AVIF все одно потребує sharp — dep залишається.

**Розглянуті альтернативи:** (1) Повністю замінити на Bun.Image — відхилено через AVIF регресію на UI. (2) Прибрати GIF-підтримку і мігрувати Bono-анімації на WebP/APNG — обговорено, не вирішено. (3) `gifsicle-wasm-browser` замість sharp для GIF — обговорено як Option C, не реалізовано.

**Зачіпає:** `npm/src/index.js` (не змінено, рішення pending), `bench/codecs/index.mjs` (hybrid logic реалізована як `bun-image-hybrid` variant), `docs/bench/2026-05-26-bun-image-vs-sharp.md`

## Knowledge Bun.Image AVIF компресія залежить від типу контенту (photo vs UI)

**Контекст:** При замірі Bun.Image vs sharp виявлено нетривіальну поведінку AVIF-кодека залежно від типу зображення.

**Рішення/Процедура/Факт:** На Kodak photo corpus (24 PNG, 768×512, nature photos) — Bun.Image AVIF дає −10% розміру відносно sharp (паритет). На UI-PNG corpus (25 PNG, flat backgrounds + logos) — **Bun.Image AVIF +38% більше** за sharp. `Bun.Image.backend === 'system'` на macOS = Apple ImageIO; AVIF-кодек у ньому (VideoToolbox/ImageIO) оптимізований під photo-content. libavif (яку використовує sharp) краще стискає flat UI через `aom`-кодек із tuned параметрами для solid color fills.

**Обґрунтування:** Не обговорювалось — емпіричне відкриття при аналізі JSON-результатів micro-bench.

**Розглянуті альтернативи:** Не обговорювались.

**Зачіпає:** Вибір кодека для AVIF у `npm/src/index.js`, `docs/bench/2026-05-26-bun-image-vs-sharp.md` (секція UI-corpus), `docs/adr/_inbox/20260526-224541-bun-image-revisited.md`
