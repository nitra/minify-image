## ADR Replace sharp → Bun.Image у @nitra/minify-image 4.0.0

**Контекст:** Пакет `@nitra/minify-image` використовував `sharp` як єдиний кодек для PNG, JPEG, AVIF, GIF. Бенчмарк на Kodak-корпусі показав, що `Bun.Image` (macOS backend = Apple ImageIO + ANE) у 9× швидший на AVIF і дає паритет по розміру; на e2e UI-корпусі — 8.9× швидший і output на 19% менший. GIF використовується лише у ~7 унікальних анімаціях (196 файлів у `~/www`, 87% < 10 KB spinners, вже оптимізовані).

**Рішення:** Версія 4.0.0 — breaking change:
- `npm/src/index.js` — замінено `.jpeg/.jpg/.png/.avif` compressors на `Bun.Image`; `writeAvif()` переписано; GIF видалено з glob-pattern, при виявленні `.gif` видається `consola.warn` і файл пропускається.
- `npm/package.json` — `sharp` видалено з `dependencies`, version `3.6.0 → 4.0.0`.
- `demo/test/run.test.js` — видалено `'minified.gif'` з `expectedFiles`; виправлено `runCli` (обхід Bun 1.3.14 bug: `NODE_ENV=test` мутує consola і stdout дочірнього процесу порожній).
- `npm/CHANGELOG.md` — секції `### Changed` / `### Removed` з метриками.

**Обґрунтування:** Повне видалення `sharp` прибирає ~17 MB disk-footprint з `node_modules` користувача (macOS arm64; на Linux більше через multi-arch binaries). `Bun.Image` вже є transitive dependency Bun ≥ 1.3 (в `engines`), тому net-cost = 0. GIF — маргінальний формат (7 унікальних файлів, рідко змінюються), а тримати native binary заради них економічно безглуздо.

**Розглянуті альтернативи:**
- **(B) Sharp лише для AVIF** — гібрид; відхилено, бо dep залишається, а AVIF на photo дав паритет (Bun.Image 0.3% більше).
- **(C) Видалити AVIF-генерацію** — відхилено, AVIF активно використовується у проектах.
- GIF warn замість silent skip — обрано, щоб CI помічав пропущені файли.

**Зачіпає:** `npm/src/index.js`, `npm/package.json`, `npm/CHANGELOG.md`, `demo/test/run.test.js`, published `@nitra/minify-image` (public API: AVIF розмір на flat UI-content ~+38% vs sharp — documented trade-off).

## Knowledge Bun 1.3.14: NODE_ENV=test глушить stdout дочірнього процесу

**Контекст:** У `bun:test` середовищі дочірні процеси, запущені через `Bun.spawn` або `node:child_process.spawnSync`, отримують `NODE_ENV=test`. `consola` в режимі `test` вимикає `info`-логи у stdout — тест-harness бачить порожній рядок замість CLI-виводу.

**Рішення/Факт:** У `runCli` хелпері тестів явно підміняємо `NODE_ENV`:
```js
const env = { ...process.env, NODE_ENV: 'production' }
```
Це відновлює повний вивід consola у spawned CLI.

**Обґрунтування:** Підтверджено ізоляційним тестом: з `NODE_ENV=test` stdout len = 0, з `NODE_ENV=production` / `TERM=xterm-256color` — повний вивід. Проблема не у `Bun.spawn` API, а у consola behavior.

**Розглянуті альтернативи:** Переключення на `execFileSync` — зайвий рефактор; явне скидання `NODE_ENV` менш інвазивне.

**Зачіпає:** `demo/test/run.test.js:runCli`, будь-який Bun-тест, що spawn-ить CLI з consola.

## ADR Bun.Image AVIF: photo vs UI-flat — різна поведінка

**Контекст:** При прийнятті рішення «sharp геть» критично знати, чи AVIF через `Bun.Image` (Apple ImageIO) завжди ефективний. Бенчмарк проведено на двох корпусах.

**Рішення/Факт:**
- **Kodak-корпус (24 photo PNG 768×512):** Bun.Image AVIF = паритет (+0.3% розміру), 9.23× швидше, ΔSSIM = −0.004 (невидима різниця).
- **UI-корпус (25 PNG 0.25–1.21 MB, flat градієнти/логотипи):** Bun.Image AVIF = **+37.8% більший файл**, 7.13× швидше. Apple ImageIO погано стискає flat UI-контент у AVIF.
- **PNG на обох корпусах:** Bun.Image виграє (Kodak −20%, UI −38%).

**Обґрунтування:** Apple ImageIO AVIF оптимізований під фото (natural image statistics), а не під solid-color/gradient UI-контент. libavif у sharp використовує AOM encoder із кращим rate-distortion на flat content.

**Розглянуті альтернативи:** Повернути sharp лише для AVIF — відхилено, бо цільові проекти використовують AVIF переважно для photo-assets, де паритет.

**Зачіпає:** `npm/src/index.js:writeAvif`, `docs/bench/2026-05-26-bun-image-vs-sharp.md`, `docs/bench/2026-05-27-kodak-png-to-avif.md`. Потрібна додаткова перевірка на Linux Docker (де backend ≠ `system`).
