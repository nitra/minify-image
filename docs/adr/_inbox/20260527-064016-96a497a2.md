## ADR Bun.Image як частковий замінник sharp: per-format рекомендація після виправленого бенчмарку

**Контекст:** Попередній ADR (`20260526-054228-bun-image-benchmark.md`) відхилив Bun.Image на основі замірів, де не враховувалися `palette: true` для PNG та `mozjpeg`/`progressive` для JPEG — що спотворювало порівняння не на користь Bun.Image. Потрібна переоцінка з коректною методологією.

**Рішення/Процедура/Факт:** Після двох незалежних замірів — Kodak photo-корпус (24 PNG, 768×512) та реальний UI-корпус (25 PNG з lumik-ru/bono, 0.25–1.21 MB) — прийнято гібридну стратегію: PNG/JPEG/WebP → Bun.Image; AVIF → sharp; GIF → sharp (Bun.Image не має GIF encoder). Harness записано в `bench/`, звіт у `docs/bench/2026-05-26-bun-image-vs-sharp.md`, ADR у `docs/adr/_inbox/20260526-224541-bun-image-revisited.md`.

**Обґрунтування:** Bun.Image виграє по PNG (−20% Kodak, −38% UI) та швидкості (2.9–8.9× E2E), але **програє на AVIF для UI-контенту (+38% більший файл)**. Apple ImageIO (`backend: system`) чудово стискає AVIF-фото, але погано обробляє flat UI fills. JPEG на Bun.Image дає +12% розмір, але SSIM +0.133 (якісніший для UI без mozjpeg). GIF: 196 файлів у `~/www/nitra`, 87% — мікро-loaders < 10 KB, лише ~7 унікальних великих анімацій; sharp залишається для GIF.

**Розглянуті альтернативи:** (1) Повний перехід на Bun.Image — відхилено: AVIF на UI гірший, GIF відсутній. (2) Залишити sharp повністю — відхилено: E2E 8.94× повільніший, занадто велика втрата. (3) Замінити GIF через `gifsicle-wasm-browser` і прибрати sharp повністю — вимагає Linux replication спочатку.

**Зачіпає:** `npm/src/index.js` (compressors PNG/JPEG/AVIF/GIF), `bench/` (весь harness), `docs/bench/2026-05-26-bun-image-vs-sharp.md`, `docs/adr/_inbox/20260526-224541-bun-image-revisited.md`

## Knowledge AVIF поведінка Bun.Image відрізняється між photo і UI контентом

**Контекст:** При плануванні переходу на Bun.Image для всіх форматів виникло питання, чи однакова перевага на photo і UI-PNG. Це не очевидно з документації Bun.Image.

**Рішення/Процедура/Факт:** На Kodak photo-корпусі Bun.Image AVIF дає паритет розміру з sharp. На реальному UI-корпусі (lumik-ru/bono logos, flat graphics) Bun.Image AVIF дає **+38% більший файл** при 7.13× прискоренні. Bun.Image на macOS використовує `backend: system` = Apple ImageIO/Media Engine — він оптимізований під природні фото (ANE hardware acceleration), але не під flat UI fills.

**Обґрунтування:** libavif (який використовує sharp) краще оптимізує flat areas з низькою варіативністю. Apple ImageIO AVIF — hardware-fast, але орієнтований на фотографічний контент. Без перевірки типу контенту не можна узагальнювати результати AVIF-бенчмарку.

**Розглянуті альтернативи:** не обговорювалися

**Зачіпає:** рішення по `@nitra/minify-image` AVIF-стратегії; будь-які проекти де AVIF застосовується до UI-скрінотів, логотипів, іконок (не до фотографій)

## ADR Методологія бенчмарку image-кодеків: Kodak + SSIM/DSSIM + corpus parity

**Контекст:** Попередні benchmark-сесії в проєкті дали неправильні висновки через відсутність parity параметрів між кодеками, бідний корпус з власних активів (вже оптимізованих), і відсутність метрик якості — порівнювали лише байти.

**Рішення/Процедура/Факт:** Стандарт для image-codec бенчмарків у цьому проєкті: (1) **Kodak Lossless True Color Image Suite** як primary corpus (24 PNG, 768×512, з `https://r0k.us/graphics/kodak/kodak/`); (2) власний UI-corpus з реальних незаторканих активів проєкту як secondary corpus; (3) **параметри адаптерів** — дзеркалити реальні CLI-параметри (`palette:true`, `quality:75`, `progressive:true`), фіксувати silently-ignored опції; (4) **метрики якості** — SSIM (`ssim.js`) + DSSIM CLI (`dssim`) для lossy форматів, декодування через нейтральний sharp.raw(); (5) **micro-bench** (N=10, відкинути перший прогін) + **E2E** (N=3, видаляти cache між прогонами). Корпуси в `.gitignore`, harness — в git.

**Обґрунтування:** Власний корпус не підходить — `@nitra/minify-image` вже стиснув усі активи в `~/www/nitra` (максимальний реальний PNG — 1.21 MB). Без parity параметрів порівняння нечесне. Без SSIM/DSSIM менший файл ≠ краща якість (lossy кодеки).

**Розглянуті альтернативи:** butteraugli — відхилено (складніше без очевидної переваги над DSSIM+SSIM). Власний корпус лише — відхилено (вже оптимізований). Тільки мікро-бенч — відхилено (може не масштабуватися до real-world E2E).
