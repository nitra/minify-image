## ADR Гібридна стратегія кодеків: Bun.Image для PNG/JPEG/WebP, sharp для AVIF
**Контекст:** Попередній ADR `20260526-054228` рекомендував залишити sharp єдиним бекендом, але мав методологічні помилки: не враховував `palette:true` для PNG (давав 68% економії на тесті), ігнорував SSIM/DSSIM, використовував нерепродуктивний корпус власних активів. Виникло питання — чи висновок змінюється при apples-to-apples порівнянні.
**Рішення/Процедура/Факт:** Впроваджено `bench/` harness (Kodak 24 PNG + 25 реальних UI-PNG з lumik-ru/bono, N=10 micro + N=3 e2e, SSIM через `ssim.js` + DSSIM через `dssim` CLI). Результат: Bun.Image виграє PNG (−38% UI, −20% Kodak, 4–7× швидше), JPEG (2.9× швидше, на UI SSIM +0.133 завдяки відсутності mozjpeg-деградації на flat fills), WebP (паритет розміру, 1.1× швидше). AVIF — інверсія: на фото паритет, на UI-контенті Bun.Image +38% більший (Apple ImageIO слабко стискає flat fills). GIF — виключено як не актуальний (файли у `~/www` є, але через `@nitra/minify-image` в реальних проектах не проходять). Рекомендація зафіксована у `docs/bench/2026-05-26-bun-image-vs-sharp.md` та `docs/adr/_inbox/20260526-224541-bun-image-revisited.md`: PNG/JPEG/WebP → Bun.Image; AVIF → sharp; GIF → можна прибрати.
**Обґрунтування:** Bun.Image (backend `system` = Apple ImageIO) використовує ANE/Media Engine на macOS arm64, звідси E2E 8.94× швидше за sharp на всьому корпусі. Для AVIF Apple ImageIO добре стискає фото (DCT-подібний контент), але гірше справляється з flat UI через відмінності в encoder tuning відносно libavif (який використовує sharp). Видалення підтримки GIF — некритична зміна, бо реального використання немає.
**Розглянуті альтернативи:** (1) Повне заміщення sharp на Bun.Image — відхилено через AVIF регресію на UI. (2) Залишити sharp повністю — відхилено через підтверджені переваги Bun.Image на трьох форматах. (3) Гібрид + залишити GIF через sharp — відхилено після підтвердження користувачем нульового GIF-usage у продакшені.
**Зачіпає:** `npm/src/index.js` (compressor block — міграція очікує Linux Docker replication), `bench/` (новий каталог із harness), `docs/bench/2026-05-26-bun-image-vs-sharp.md`, `docs/adr/_inbox/20260526-224541-bun-image-revisited.md`, попередній ADR `20260526-054228-bun-image-benchmark.md` (спростовано)

## Knowledge AVIF-якість Bun.Image залежить від типу контенту: фото vs UI
**Контекст:** Під час benchmark-сесії Bun.Image (macOS, `Bun.Image.backend === 'system'` = Apple ImageIO) показав суперечливі результати для AVIF на двох корпусах.
**Рішення/Процедура/Факт:** На Kodak photo corpus (24 PNG, 768×512, природні сцени) AVIF Bun.Image ≈ паритет з sharp за розміром при схожому SSIM. На UI-corpus (25 PNG, lumik-ru bg + bono logos — flat fills, градієнти, лого) Bun.Image AVIF на **+38% більший** ніж sharp при нижчому SSIM (−0.005). Sharp використовує libavif з rav1e/libaom; Apple ImageIO використовує власний AVIF encoder, оптимізований для фото-контенту.
**Обґрунтування:** Apple ImageIO AVIF encoder нe має аналога `crf`/`effort` tuning для flat/synthetic content, тоді як libavif через sharp краще використовує intra-prediction для однотонних регіонів.
**Розглянуті альтернативи:** не обговорювалися
**Зачіпає:** Будь-який код, що вирішує замінити sharp на Bun.Image для AVIF-кодування; `npm/src/index.js` (AVIF compressor блок — залишити на sharp); `docs/bench/2026-05-26-bun-image-vs-sharp.md` (зафіксовано в звіті)
