## ADR Залишити sharp як єдиний бекенд стискання (відмова від Bun.Image)

**Контекст:** Bun 1.3.14 додав вбудований `Bun.Image` API з підтримкою PNG, JPEG, WebP та AVIF без зовнішніх залежностей. Виникло питання: чи може він замінити `sharp` у `@nitra/minify-image` і спростити дерево залежностей.

**Рішення/Процедура/Факт:** Проведено замір на корпусі з 23 файлів (PNG та JPEG з `demo/test/files/` та реальних активів з `~/www/nitra`). Результати порівняння sharp (`compressionLevel:9, effort:10` для PNG; `quality:85` для JPEG) vs `Bun.Image` (`quality:100` для PNG; `quality:85` для JPEG):

**PNG:**

- Для малих PNG (≤55 KB) sharp у **2–3× швидше** (0.6–2.2 ms vs 1.2–6.7 ms); розмір виходу однаковий.
- Для великих PNG (1–20 MB) sharp у **10–15× швидше** (8–217 ms vs 109–2185 ms); `Bun.Image` видає файли **на 17–33% більші** через відсутність еквіваленту `compressionLevel`.
- Приклад: `icon.png` (19 MB) — sharp 217 ms / 1874 KB, Bun.Image 2185 ms / 2193 KB.

**JPEG:**

- Для великого JPEG (~1.2 MB) `Bun.Image` **на 41% швидше** (162 ms vs 275 ms), але виходить на 5% більше.
- Для малого JPEG (~8 KB) sharp **у 3.2× швидше** (3.9 ms vs 12.7 ms).

**Висновок:** `Bun.Image` не є рівноцінною заміною — суттєво програє за PNG-стисканням (де більшість активів проєкту) і не підтримує GIF-кодування. `sharp` залишається єдиним бекендом.

**Обґрунтування:** Основний робочий вантаж `@nitra/minify-image` — PNG-активи Tauri (великі, 1–20 MB). Для них `Bun.Image` дає 10–15× гірший throughput і при цьому більший вихідний файл. Поки `Bun.Image` не отримає еквівалент `compressionLevel`/`effort` для PNG і не закриє розрив у продуктивності — заміна економічно невиправдана.

**Розглянуті альтернативи:**

- Повна заміна `sharp` → `Bun.Image` — відхилено (підтверджено замірами).
- Гібридний режим: `Bun.Image` для JPEG, `sharp` для PNG — ускладнює код і підтримку двох залежностей; виграш лише на великих JPEG не компенсує. Не прийнято.

**Зачіпає:** `npm/src/index.js`, `npm/package.json` (залежність `sharp`), `bench/bench.mjs` (новий бенчмарк-скрипт), `bench/results-2026-05-26T05-42-28.{json,md}` (збережені дані замірів).
