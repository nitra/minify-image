## Дизайн: бенчмарк Bun.Image vs sharp

### 1. Структура

```
demo/bench/
corpus/           ← symlink / rclone-download до kodak-suite + публічних SVG/GIF
bench-micro.js    ← мікробенч (Bun-native)
bench-e2e.js      ← CLI end-to-end порівняння
report.js         ← зводить результати у markdown
```

Жодних нових prod-deps; тільки dev/bench-only.

---

### 2. Корпус (варіант C)

| Джерело                           | Формат                   | Кількість | Мета                                    |
| --------------------------------- | ------------------------ | --------- | --------------------------------------- |
| Kodak Suite (24 фото 768×512)     | JPEG                     | 24        | якість + SSIM/butteraugli               |
| Wikimedia Commons public CC0 pack | PNG (lossy/lossless mix) | ~20       | PNG deflate + AVIF-out                  |
| GIF (опенсорс, animated/static)   | GIF                      | ~10       | лише розмір — Bun.Image без GIF encoder |
| Micro SVG (icon corpus, OpenMoji) | SVG                      | ~30       | розмір, SVGO vs no-op                   |

Завантаження через `bench-corpus.sh` (curl + unzip).

---

### 3. Метрики якості

- **JPEG / WebP / AVIF**: SSIM (через `ssim.js`, чистий JS) + butteraugli-compat через `dssim` CLI ([C бінарка](https://github.com/kornelski/dssim), homebrew install)
- **PNG** (lossless): тільки розмір + чи `decoded_pixels_match`
- **GIF**: тільки розмір (Bun.Image не підтримує GIF encode — gap явно логується)
- **SVG**: тільки розмір (SVGO обидва не торкаються)

Всі порівняння «при однаковому quality Q» (наприклад, Q=75 та Q=90 для JPEG).

---

### 4. Мікробенч

```js
// bench-micro.js
for (const file of corpus) {
  const input = await Bun.file(file).bytes()
  // SHARP variant
  const t0s = Bun.nanoseconds()
  const sharpOut = await sharp(input).jpeg({ quality: 80 }).toBuffer()
  const dtSharp = Bun.nanoseconds() - t0s
  // BUN.IMAGE variant
  const t0b = Bun.nanoseconds()
  const bunOut = await new Bun.Image(input).jpeg({ quality: 80 }).bytes()
  const dtBun = Bun.nanoseconds() - t0b
  // → запис у jsonl
}
```

Кожен файл × N=10 прогонів (warm), беремо медіану; перший прогін як warm-up (не враховується).

---

### 5. End-to-end

Два варіанти CLI в тому ж monorepo:

- `npm/src/index.js` (current, sharp)
- `npm/src/index.bun.js` (fork: sharp → Bun.Image, однаковий публічний API)

Виміряємо через `/usr/bin/time -v` (Linux) або `time` + process.hrtime (macOS). Паралелізм той самий (AVIF pool).

---

### 6. Очікувані gap-и (попередній аналіз)

|                   | sharp  | Bun.Image               |
| ----------------- | ------ | ----------------------- |
| GIF encode        | ✓      | ✗ (повернемо як is-gap) |
| HEIC encode       | ✗      | ✓                       |
| Streaming         | ✓      | ✗ (тільки buffer)       |
| Dependency weight | ~70 MB | 0 MB (built-in)         |

## ADR Benchmark Bun.Image vs sharp

**Контекст:** Пакет `@nitra/minify-image` використовує `sharp` для стискання PNG/JPEG/GIF/AVIF. Bun 1.3.14 містить вбудований `Bun.Image` з підтримкою JPEG/PNG/WebP/AVIF/HEIC без жодних нативних залежностей (~70 MB економії), що відкриває потенціал заміни sharp на вбудований кодек.

**Рішення/Процедура/Факт:** Вирішено створити окремий дослідницький (spike) харнес у `demo/bench/` для порівняння `sharp` та `Bun.Image` по трьох осях: швидкість кодування (мікробенч через `Bun.nanoseconds()` + end-to-end CLI прогін), якість стискання (SSIM через `ssim.js` + DSSIM через `dssim` CLI при `quality=75` і `quality=90`), функціональні gap-и. Корпус — публічний бенчмарк-набір: 24 Kodak JPEG, ~20 PNG CC0, ~10 GIF CC0, 30 OpenMoji SVG. Буде написано форк CLI `npm/src/index.bun.js` з ідентичним публічним API, де `sharp` замінено на `Bun.Image`. Результати — у `demo/bench/results/bench-results.md`. Кінцеве рішення (замінити чи ні) — після аналізу числових результатів, зафіксоване в ADR.

**Обґрунтування:** Bun.Image вже вбудований у Bun 1.3.14 (підтверджено `typeof Bun.Image === 'function'`), тому якщо якість і швидкість паритетні — заміна sharp прибирає ~70 MB нативної залежності та спрощує CI/CD. Для GIF encode паритету немає (метод відсутній) — цей gap треба кількісно задокументувати. Без замірів рішення про заміну не може бути обґрунтованим.

**Розглянуті альтернативи:**

- Виміряти лише розмір файлу (без SSIM/DSSIM) — відхилено: менший файл при гіршій якості не є перемогою
- Лише мікробенч без end-to-end — відхилено: ризик пропустити накладні витрати ініціалізації Bun.Image
- Тільки end-to-end без мікробенчу — відхилено: губиться роздільна здатність по форматах
