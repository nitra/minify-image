## ADR Дизайн бенчмарку sharp vs. Bun.Image для `@nitra/minify-image`

**Контекст:** Виник запит перевірити, чи можна замінити `sharp` на `Bun.Image` (з'явився у Bun 1.3.14) як кодек у `@nitra/minify-image`, не жертвуючи якістю стискання та швидкодією.

**Рішення/Процедура/Факт:**
Заплановано двоетапний замір у `bench/`:

1. **Мікро-бенч** — `Bun.nanoseconds()` навколо одного encode-виклику, медіана з N прогонів для кожного кодека (PNG, JPEG, AVIF, WebP). Corpus: **Kodak Lossless True Color Image Suite** (24 PNG, 768×512, ~18 MB), завантажується у `bench/corpus/kodak/`.
2. **End-to-end CLI** — повний прогін `@nitra/minify-image` по корпусу двічі: поточна реалізація через `sharp` і форк з `Bun.Image`.

Якість вимірюється трьома методами залежно від типу:

- **JPEG / AVIF / WebP**: розмір файла + SSIM + butteraugli/DSSIM (чистий JS, без бінарок)
- **PNG (lossless)**: тільки розмір (SSIM не релевантний)
- **GIF / SVG**: поза областю — `Bun.Image` не має GIF-кодера; SVG не є растровим кодеком

Виявлено під час зондування API: `Bun.Image` на macOS використовує `backend: system` (ImageIO). Параметри `mozjpeg: true` та `effort: 10` приймаються, але **не впливають на результат** (silently ignored) — важливо для fairness JPEG-порівняння.

**Обґрунтування:** Тільки end-to-end без мікро-бенчу губить роздільну здатність по форматах; тільки мікро-бенч не показує, чи масштабуються результати на реальний user-perceived час. SSIM без butteraugli/DSSIM недостатній для lossy-кодеків, де менший файл ≠ краща якість. Kodak Suite — академічний стандарт для codec-бенчмарків, покриває реальний фото-контент.

**Розглянуті альтернативи:**

- Корпус: `demo/test/files/` (забракований — замало для статистики), власні файли з `~/www/nitra/`, публічний набір (**обрано**)
- Якість: лише розмір файла (забраковано), розмір + SSIM (відхилено на користь повного пакета), розмір + SSIM + butteraugli/DSSIM (**обрано**)
- Швидкодія: тільки мікро-бенч (відхилено), тільки end-to-end (відхилено), обидва (**обрано**)
