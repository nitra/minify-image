## ADR Заміна sharp на Bun.Image в @nitra/minify-image 4.0.0

**Контекст:** Bun 1.3.14 додав вбудований `Bun.Image` API без нативних залежностей. Було прийнято рішення провести методологічно строгий benchmark на Kodak-suite (24 PNG, N=10) та реальному UI-корпусі (25 PNG з nitra-проектів) з метрикою розміру + SSIM/DSSIM, щоб вирішити чи можна прибрати `sharp` як runtime dependency з опублікованого пакету.

**Рішення/Процедура/Факт:** `sharp` видалено з `dependencies` у `npm/package.json`. `npm/src/index.js` переписано: PNG/JPEG/AVIF кодуються через `new Bun.Image(buf).png/jpeg/avif().bytes()`. GIF-підтримка видалена: `.gif` файли skip-куються з `consola.warn`. Версія пакету піднята до `4.0.0` (breaking). `engines.node` прибрано — пакет Bun-only. `node_modules` зменшились з 48 MB → 33 MB (−15 MB, −3 native packages: `sharp`, `@img/sharp-darwin-arm64`, `@img/sharp-libvips-darwin-arm64`).

**Обґрунтування:** Бенчмарк показав:
- **PNG:** −20–38% менший файл, 2–4× швидше (паritет→win по всіх корпусах)
- **JPEG:** +4–12% більший файл, але 2.87–3.37× швидше; SSIM +0.133 на UI (якість краща)
- **AVIF (photo/Kodak):** +0.3% розміру (паритет), **9.23× швидше** завдяки Apple ImageIO+ANE на macOS arm64
- **AVIF (UI flat-graphics):** +37.8% розміру — регресія; прийняте рішення приймати цей trade-off, оскільки UI-AVIF у production-проектах сумарно < 500 KB, а net-gain від відсутності native build значний
- GIF: 190 файлів на ~33 MB у `~/www/nitra`, але 87% < 10 KB, а унікальних великих (~7 файлів) — маргінальний use case; cost тримати `sharp` лише заради них перевищує benefit
- E2E end-to-end CLI: **8.94× швидше, −19.1% вихідний розмір**
- Platform: macOS arm64, Bun 1.3.14, `Bun.Image.backend = system`; Linux результати не верифіковано

**Розглянуті альтернативи:**
- **(B) Гібрид: sharp залишається лише для AVIF+GIF** — відхилено: зберігає native build dep, ускладнює install для CLI-users
- **(C) Прибрати AVIF-генерацію разом із GIF** — відхилено: `--avif` активно використовується в production workflows
- **Migrate GIF → gifsicle wasm** — відхилено: ще одна dep заради маргінального use case; рекомендовано migrate на WebP/APNG

**Зачіпає:** `npm/src/index.js`, `npm/package.json` (version 3.6.0→4.0.0, deps), `npm/CHANGELOG.md`, `demo/test/run.test.js` (видалено `minified.gif` з `expectedFiles`)

## ADR Методологія benchmark Bun.Image vs sharp

**Контекст:** Попередній ADR (`docs/adr/_inbox/20260526-054228-bun-image-benchmark.md`) зафіксував «sharp залишається» на основі замірів без `palette: true` для PNG, без `progressive` для JPEG, і без SSIM/DSSIM — тобто не apples-to-apples. Різниця між методологіями виявила протилежний висновок для PNG та AVIF.

**Рішення/Процедура/Факт:** Побудовано harness у `bench/` зі стандартизованими adapter-ами (`bench/codecs/sharp.mjs`, `bench/codecs/bun-image.mjs`) з однаковими параметрами (`compressionLevel:9, palette:true` для PNG; `quality:75, progressive:true` для JPEG; `quality:40` для AVIF). Корпус: Kodak Lossless True Color Image Suite (24 PNG, 768×512) + UI-corpus (25 PNG з реальних nitra-проектів). Якість: SSIM через `ssim.js` + DSSIM через `dssim` CLI (Kornelski). Швидкість: N=10 прогонів, відкидання першого (JIT warmup), median + p95. E2E: 3 прогони CLI на `bench/e2e-corpus/`, медіана wall-clock.

**Обґрунтування:** Попередній замір тестував `sharp` з `mozjpeg:true, palette:true` проти `Bun.Image` без цих опцій — тому Bun.Image виглядав гіршим по розміру. `effort:10` і `mozjpeg:true` silently ігноруються Bun.Image 1.3.14 (`Bun.Image.backend = system`). Фіксація цього факту в звіті дозволила чесно порівнювати.

**Розглянуті альтернативи:**
- **butteraugli** як метрика якості — відхилено: складніша інтерпретація, SSIM+DSSIM достатньо
- **DSSIM-only** без SSIM — відхилено: потрібна bінарка, fallback на SSIM-only якщо нема
- **Власний UI-корпус замість Kodak** — додано як **другий** корпус (UI): Kodak — академічний стандарт, UI — реальний use case; обидва разом дали повну картину

**Зачіпає:** `bench/` (весь harness: `download-corpus.mjs`, `codecs/`, `quality.mjs`, `micro.mjs`, `e2e.mjs`, `report.mjs`), `docs/bench/2026-05-26-bun-image-vs-sharp.md`, `docs/bench/2026-05-27-kodak-png-to-avif.md`

## Knowledge AVIF на UI flat-graphics: Bun.Image (Apple ImageIO) vs sharp (libavif)

**Контекст:** AVIF-кодек `Bun.Image.backend = system` на macOS arm64 використовує Apple ImageIO. При `quality:40` на photo-корпусі (Kodak) він дає паритет по розміру (+0.3%) і 9.23× speedup. На UI flat-graphics (логотипи, фони з градієнтами і плоскими зонами) той самий `quality:40` дає **+37.8% більший файл** ніж sharp/libavif.

**Рішення/Процедура/Факт:** Різниця пояснюється архітектурою кодеків: libavif (sharp) оптимізовано для flat/synthetic content через кращий intra-prediction і rate-distortion optimization для однотонних зон. Apple ImageIO AVIF оптимізовано для photo через ANE hardware-acceleration. Це не баг — це design trade-off Apple ImageIO vs libavif.

**Обґрунтування:** У context `@nitra/minify-image` UI-AVIF використовується рідко і сумарний обсяг small (< 500 KB), тому +37.8% прийнятний. Для photo-intensive проектів (Kodak-like) Bun.Image AVIF строго кращий (9× швидше, паритет якості).

**Розглянуті альтернативи:** не обговорювались; є follow-up завдання «Linux Docker replication» для перевірки на linux backend.

**Зачіпає:** `npm/src/index.js` (AVIF encoding path), `docs/bench/2026-05-26-bun-image-vs-sharp.md` (UI-corpus section)
