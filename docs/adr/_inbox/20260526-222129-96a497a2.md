## ADR Методологія бенчмарку Bun.Image vs sharp: Kodak-корпус + SSIM/DSSIM

**Контекст:** Попередній ADR (`20260526-054228-bun-image-benchmark.md`) зробив висновок «залишити sharp», але методологія мала три суттєві вади: PNG тестувалось без `palette: true` (а саме ця опція дає основну економію), якість порівнювалась лише за розміром без метрик спотворення, корпус складався з 23 власних активів без можливості відтворення.
**Рішення/Процедура/Факт:** Переоцінка з виправленою методологією: корпус — Kodak Lossless True Color Suite (24 PNG 768×512, академічний стандарт); параметри кодеків — «apples-to-apples» (sharp: `quality:70, mozjpeg:true, compressionLevel:9, effort:10, palette:true`; bun-image: ті самі де підтримуються, відхилення фіксуються у звіті); метрики якості — SSIM (`ssim.js`) + DSSIM (CLI) для lossy форматів, розмір — для всіх; два рівні: per-codec мікро-бенч (10 прогонів, медіана, p95) + end-to-end CLI (3 прогони, медіана, RSS). Весь harness збережено у `bench/`, результати відтворюються однією командою. Звіт — `docs/bench/2026-05-26-bun-image-vs-sharp.md`.
**Обґрунтування:** «Менший файл без контролю якості» — нечесне порівняння: AVIF може бути на 19% меншим саме через нижчу якість. Без Kodak-корпусу результат не відтворюваний іншою командою. `palette:true` змінює PNG-результат з 32 KB до 10 KB — без нього порівняння PNG безглузде.
**Розглянуті альтернативи:** (1) Власні активи з `~/www/nitra/` — відхилено як нерепродуковані. (2) butteraugli — відхилено на користь DSSIM (simpler, однозначна інтерпретація). (3) Тільки end-to-end — відхилено, бо губить роздільну здатність по форматах.
**Зачіпає:** `bench/` (новий), `docs/bench/2026-05-26-bun-image-vs-sharp.md`, `docs/superpowers/specs/2026-05-26-bun-image-vs-sharp-benchmark-design.md`, `docs/superpowers/plans/2026-05-26-bun-image-vs-sharp-benchmark.md`

## Knowledge Bun.Image silently ігнорує `mozjpeg`, `effort` та `progressive` (Bun 1.3.14, macOS)

**Контекст:** При переносі параметрів кодування з sharp на Bun.Image виявилось, що кілька ключових опцій приймаються API без помилок, але не впливають на результат.
**Рішення/Процедура/Факт:** На Bun 1.3.14 з `backend: system` (macOS ImageIO): `mozjpeg: true` — silently ignored, `effort: N` (для PNG) — silently ignored, `progressive: true` (JPEG) — silently ignored. Перевірено empirically: `sharp(buf).png({ effort: 10 })` → 32 242 bytes; `new Bun.Image(buf).png({ effort: 10 })` → 32 242 bytes; `new Bun.Image(buf).png({ compressionLevel: 9, palette: true })` → 10 792 bytes. Тобто `palette: true` працює, `effort` — ні.
**Обґрунтування:** `backend: system` означає делегування до macOS ImageIO, який не підтримує ці параметри. На Linux backend буде інший (libvips або інше) — поведінка може відрізнятись.
**Розглянуті альтернативи:** не обговорювались.
**Зачіпає:** `bench/codecs/bun-image.mjs`, будь-який код що використовує `Bun.Image` для JPEG/PNG і розраховує на `mozjpeg`/`effort`/`progressive`

## Knowledge Результати бенчмарку Bun.Image vs sharp (Kodak, macOS arm64, Bun 1.3.14)

**Контекст:** Після виправленої методології (Kodak, apples-to-apples, SSIM) отримано кількісні результати для прийняття рішення щодо заміни sharp у `@nitra/minify-image`.
**Рішення/Процедура/Факт:** Мікро-бенч (медіана по 24 зображеннях): PNG — bun-image 1.7ms vs sharp 1459ms (862x швидше), однаковий розмір; AVIF — bun-image 1.9ms vs sharp 1061ms (558x швидше), bun-image на 19% менший файл але SSIM -0.015 (нижча якість); WebP — bun-image 4.4ms vs sharp 135ms (31x швидше), але bun-image на 38% більший файл; JPEG — bun-image 21.4ms vs sharp 9.6ms (2.2x повільніше), bun-image на 4% менший файл, SSIM +0.010 (вища якість). End-to-end (25 файлів): bun-image 214ms vs sharp 34939ms (163x швидше), RSS: 182MB vs 203MB. DSSIM недоступний (потребує `brew install dssim`) — SSIM-only.
**Обґрунтування:** Різниця PNG/AVIF/WebP пояснюється тим, що sharp використовує `effort:10` та libvips (CPU-intensive), Bun.Image — macOS ImageIO (апаратно-прискорений). JPEG повільніший у bun-image, бо sharp використовує mozjpeg (хоча ця опція на Bun.Image ігнорується — JPEG-різниця пояснюється різними кодеками підсистеми).
**Розглянуті альтернативи:** не обговорювались.
**Зачіпає:** `npm/src/index.js` (потенційна заміна компресорів), `docs/adr/_inbox/20260526-054228-bun-image-benchmark.md` (попередній ADR потребує переоцінки з урахуванням цих даних)
