## ADR Bun.Image vs sharp — бенчмарк з виправленою методологією (Kodak + UI-корпус, SSIM/DSSIM, palette parity)

**Контекст:** Попередній ADR (`20260526-054228-bun-image-benchmark.md`) відхилив Bun.Image через нечесне порівняння: PNG без `palette:true`, JPEG без `mozjpeg`/`progressive`, тільки свої активи (не репродуковано), без метрик якості (лише байти). Виникла потреба в коректному повторному вимірі.

**Рішення/Процедура/Факт:** Побудовано standalone-харнес у `bench/` з двома корпусами (Kodak 24 PNG академічний стандарт + UI-PNG з реальних проектів nitra), чотирма adapter-варіантами (`sharp`, `bun-image`, `sharp-default`, `bun-image-default`), SSIM (`ssim.js`) і DSSIM (Kornelski CLI), micro-bench (N=10, медіана, p95) і end-to-end CLI-прогін (N=3, медіана wall-clock). Bun.Image параметри дзеркалять поточний CLI: `palette:true`, `compressionLevel:9` для PNG; `quality:75, progressive:true` для JPEG; `quality:40` для AVIF. Явно зафіксовано: `mozjpeg:true` і `effort:10` — silently ignored у Bun.Image.

**Обґрунтування:** Виправлено три методологічних помилки попереднього ADR: (1) `palette:true` для PNG (дає ~70% економію на UI-PNG), (2) апples-to-apples параметри кодеків, (3) стандартизований академічний корпус (Kodak) замість власних артефактів. Платформа зафіксована: macOS arm64, `Bun.Image.backend === 'system'` (Apple ImageIO + ANE/Media Engine hardware acceleration).

**Розглянуті альтернативи:** Тестування лише на `demo/test/files/` (відхилено: занадто мало для статистики), butteraugli як метрика якості (відхилено: SSIM+DSSIM достатньо), Linux Docker replication (відкладено як follow-up).

**Зачіпає:** `bench/` (новий каталог), `docs/bench/2026-05-26-bun-image-vs-sharp.md`, `docs/bench/2026-05-27-kodak-png-to-avif.md`, `docs/adr/_inbox/20260526-224541-bun-image-revisited.md`, `docs/superpowers/specs/2026-05-26-bun-image-vs-sharp-benchmark-design.md`, `docs/superpowers/plans/2026-05-26-bun-image-vs-sharp-benchmark.md`

## ADR Диференційована стратегія для кожного формату (Bun.Image partial migration)

**Контекст:** За результатами бенчмарку виявилось, що Bun.Image не має однаково переконливих показників по всіх форматах: перевага залежить від типажу контенту (photo vs. UI-flat).

**Рішення/Процедура/Факт:** Рекомендована стратегія (зафіксована у звітах і оновленому ADR): PNG → Bun.Image (−20–38% розміру, 2–4× швидше для всіх типажів контенту); JPEG → Bun.Image (швидше, на UI +0.133 SSIM через відсутність mozjpeg-перекомпресії); WebP → Bun.Image (паритет розміру, незначний speedup); **AVIF photo-корпус → Bun.Image** (9.23× швидше, паритет байт); **AVIF UI-контент → sharp** (+38% розміру у Bun.Image на flat-графіку); GIF → sharp (Bun.Image не має encoder). Sharp залишається в deps для AVIF UI і GIF.

**Обґрунтування:** Bun.Image AVIF використовує Apple ImageIO hardware encoder — чудово для photo (де encode efficiency High), слабо для flat UI-контент (де apple encoder не оптимізований під solid-fills). Різниця між корпусами: Kodak photo → паритет (+0.3%); реальні UI-PNG lumik-ru/bono → +38% у Bun. GIF: 196 файлів у `~/www` (41 MB), 87% < 10 KB (loaders), 7 унікальних > 1 MB (Bono animations) — недостатньо, щоб тримати sharp тільки заради GIF якщо AVIF можна вирішити інакше.

**Розглянуті альтернативи:** Повний remove sharp (відхилено: AVIF UI regression, GIF gap); окремий `gifsicle-wasm` для GIF (відкладено); вилучити GIF-підтримку з CLI і мігрувати до WebP/APNG (розглядається як follow-up); Linux Docker replication перед production-migration (обов'язковий крок перед злиттям в main).

**Зачіпає:** `npm/src/index.js` (при прийнятті — заміна compressor-блоку), `npm/package.json` (sharp залишається або умовний), поведінка AVIF-виводу для кінцевих споживачів пакета.
