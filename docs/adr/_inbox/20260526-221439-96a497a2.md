## ADR Бенчмарк sharp vs. Bun.Image: рішення залишити sharp у @nitra/minify-image

**Контекст:** Bun 1.3.14 додав вбудований `Bun.Image` API з підтримкою PNG, JPEG, WebP, AVIF, HEIC, що відкрило можливість прибрати залежність від `sharp` (libvips) у `@nitra/minify-image`. Потрібно було виміряти реальний trade-off між швидкодією та якістю стискання.

**Рішення/Процедура/Факт:** Розроблено та виконано бенчмарк-сюїту (`docs/adr/_bench/`) на корпусі 31 файл (Kodak 24×PNG, 5 USC/SIPI PNG, 2 GIF, 2 SVG) з 40 прогонами на пару файл/формат. Інструменти: `ssim.js` (pure-JS SSIM) + `dssim` CLI (DSSIM). Результати: Bun.Image у 11.4× швидша end-to-end (35 773 ms → 3 128 ms), для AVIF — у 32× швидша; проте видає файли у 2–3× більші (AVIF +177%, WebP +122%, JPEG +28%) при нижчому SSIM (JPEG: 0.946 vs 0.959) та вищому DSSIM. GIF encoder у Bun.Image відсутній. Прийнято рішення **залишити sharp** — `npm/src/index.js` та `npm/package.json` не змінено.

**Обґрунтування:** Мета бібліотеки — мінімізація розміру зображень, тому перевага Bun.Image у швидкодії не компенсує суттєво більші вихідні файли та гіршу якість для lossy-форматів. Відсутність GIF encoder є блокером для збереження існуючого функціоналу. Додатковий ризик: `Bun.Image` на macOS використовує `system` backend (CoreGraphics/ImageIO), що означає платформо-залежні результати — на Linux поведінка може відрізнятися.

**Розглянуті альтернативи:** Повна заміна sharp → Bun.Image (відхилено: більший розмір, гірша якість, немає GIF); часткова заміна лише для AVIF/WebP encode (відкладено до моменту, коли Bun.Image покращить якість lossy-кодування або з'явиться режим «fast preview»); залишити sharp (прийнято).

**Зачіпає:** `npm/src/index.js`, `npm/package.json` (залежність `sharp`), `docs/adr/_bench/` (новий бенчмарк-харнес), `docs/adr/_inbox/20260526-222801-sharp-vs-bun-image.md` (ADR з повними таблицями результатів).
