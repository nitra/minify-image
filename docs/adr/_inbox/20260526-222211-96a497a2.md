## ADR Методологічно коректний бенчмарк Bun.Image vs sharp

**Контекст:** Bun 1.3.14 додав вбудований `Bun.Image` API. Попередній ADR (`docs/adr/_inbox/20260526-054228-bun-image-benchmark.md`) зафіксував «sharp залишається», але мав критичні методологічні помилки: PNG тестувався без `palette: true` (дає ~3× економію), JPEG — без `mozjpeg`/`progressive`, виміряно лише байти без якісних метрик (SSIM/DSSIM), корпус — власні активи, не репродуковано.

**Рішення/Процедура/Факт:** Написано дизайн (`docs/superpowers/specs/2026-05-26-bun-image-vs-sharp-benchmark-design.md`) та план реалізації (`docs/superpowers/plans/2026-05-26-bun-image-vs-sharp-benchmark.md`) для 11-задачного harness у директорії `bench/`. Параметри кодеків дзеркалять поточний CLI (`palette: true`, `compressionLevel: 9` для PNG; `quality: 75, progressive: true` для JPEG; `quality: 40` для AVIF). Корпус — Kodak Lossless True Color Suite (24 PNG, 768×512). Метрики: розмір + SSIM (`ssim.js`) + DSSIM (бінарка). Швидкість: per-codec мікробенч (10 прогонів, медіана, `Bun.nanoseconds()`) + end-to-end CLI прогін (wall-clock + maxRSS). GIF і SVG виключені — `Bun.Image` не має GIF-кодера. Результати до git не комітяться, harness — комітиться для репродукованості.

**Обґрунтування:** `Bun.Image.backend === 'system'` на macOS — це Apple ImageIO, тому `mozjpeg` і `effort` silently ігноруються. Без `palette: true` PNG-порівняння некоректне. Без SSIM/DSSIM порівняння байтів нечесне (менший файл при гіршій якості ≠ перевага). Kodak Suite — академічний стандарт, репродукований і незалежний від проєктних активів.

**Розглянуті альтернативи:** (A) Corpus = `demo/test/files/` (відхилено — 1 JPEG замало для висновків); (B) Корпус = власні активи `~/www/nitra/` (відхилено — не репродуковано); розмір-only без якості (відхилено — нечесне для lossy); butteraugli поверх SSIM+DSSIM (відхилено — зайва складність без додаткової інформації).
